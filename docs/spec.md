## 0. 需求边界更新（2026-06-20）

当前实现优先对齐 **需求A**。需求B 只做框架设计预留，不进入本阶段生产代码范围。

### 需求A：当前实现目标

1. **评价管理**
   - 来源：拼多多商家后台【商品管理】-【评价管理】。
   - 处理范围：仅处理最近 72 小时内新产生的评价；超过 72 小时、未来时间或缺失发布时间的评价必须跳过并记录审计。
   - 1-3 星评价：按 AI/规则匹配话术举报。
   - 4-5 星评价：按统一模板回复。
   - 互动隐藏：进入【评价管理】-【查看全部互动】，页面筛选近 30 天，代码层仅识别近 7 日内负面互动并生成隐藏建议，运营审批后再点击行级【隐藏评论】。

2. **店铺健康度监控**
   - 来源：拼多多商家后台【数据中心】-【服务数据】。
   - 综合体验星级：采集店铺综合体验星级、领航员综合分行业排名；领航员表格中的未达标项需要结构化保存并展示，字段包含维度、考核指标、店铺表现、是否达标、下一星级标准。
   - 售后数据：重点采集并展示纠纷退款率、平台介入率、品质退款率、平均退款时长，并基于历史数据输出趋势结论：上升、下降、平稳。
   - 评价数据：正式采集并展示店铺评价分排名、近 30 天评价数，并输出评价排名和评价条数趋势结论：上升、下降、平稳。
   - 消费者体验指标：展示消费者服务体验分、服务态度体验分、基础服务体验分、商品服务体验分、发货服务体验分、物流服务体验分，并展示平台给出的较前 7 日 +/- 百分比。

### 需求B：本阶段只做设计预留

1. **商品体检**
   - 来源：拼多多商家后台【商品管理】-【商品体检】。
   - 设计目标：采集今日体检分、待修改商品数量、商品标题、商品 ID、体检问题、影响内容、30 日销量，并由 AI/规则输出风险预警。
   - 建议数据模型：`product_check_reports` 保存店铺、日期、体检分、待修改数量、摘要；`product_check_items` 保存商品级问题明细、销量和风险判断。
   - 建议 UI：商品体检页展示总分卡片、待修改数量、风险分布、问题商品表格。

2. **店铺违规**
   - 来源：拼多多商家后台【店铺管理】-【违规信息】。
   - 店铺违规管理：按违规通知时间采集最近一周违规通知，按违规类型展示，并由 AI/规则判断高危违规。
   - 售后服务违规：选择最近 7 天，采集违规内容、处理措施、处理状态、违规时间，并由 AI/规则输出风险提醒。
   - 建议数据模型：`violation_notices` 保存来源、违规类型、内容、处理措施、处理状态、通知时间、违规时间、AI/规则风险等级。
   - 建议 UI：店铺违规页使用“店铺违规 / 售后服务违规”双标签页，展示 7 日时间线、风险分布和违规明细表。

3. **商品推广**
   - 来源：拼多多商家后台【推广平台】-【商品推广】。
   - 当前需求待定，仅保留后续扩展入口，不进入本阶段采集和 UI 实现。

## 一、项目背景

拼多多多店铺（30+）每日巡店操作当前为纯手动，日均耗费约半天。目标：用 AI + 浏览器自动化替代人工，最终输出各店铺每日巡店结果。

### 核心操作范围

| 模块 | 操作 | 类型 |
|------|------|------|
| 评价管理 | 最近 72 小时评价：1-3 星生成举报建议 / 4-5 星按模板回复 / 【查看全部互动】生成负面互动隐藏建议；举报和隐藏默认先审批后执行 | 写操作 |
| 店铺健康度 | 监控综合体验星级、领航员未达标项、售后 4 核心指标、评价数据及趋势 | 读操作 |
| 消费者体验 | 跟踪 6 项体验分，并展示平台给出的较前 7 日 +/- 百分比 | 读操作 + 记录 |

## 二、系统架构

**选型: 任务队列 + 浏览器 Worker 池（方案 B）**

### 组件

| 组件 | 技术 | 职责 |
|------|------|------|
| Web 前端 | React + Tailwind + ECharts | 日报总览、单店详情、模板管理、店铺配置 |
| API 服务层 | Fastify | REST API、用户认证、店铺管理 |
| 任务调度器 | BullMQ Repeatable Jobs | 定时触发（每日 8:00）、任务队列管理、失败重试 |
| 浏览器 Worker | Playwright × N | 登录 PDD 后台、执行巡店操作、截图留证 |
| 数据存储 | SQLite(起步)→PG + Redis + 本地文件 | 主库、队列/缓存、截图 |

### 核心流程

```
定时触发 (8:00) → 每店铺创建巡店作业 → BullMQ 队列 → Worker 池并行消费
→ 各 Worker 独立开浏览器 → 四阶段流水线 → 写结果 → 聚合日报 → Web 展示
```

### 进程清单

| 进程 | 数量 | 说明 |
|------|------|------|
| Docker | 1 | Redis + (可选) PostgreSQL |
| Scheduler | 1 | 定时触发任务 |
| Worker | N (可配) | Playwright 浏览器自动化 |
| Server | 1 | Fastify API + 前端静态文件 |

## 三、单店巡店流程（Worker 四阶段流水线）

### 阶段 1: 登录 & 进入后台
- 打开 PDD 商家登录页
- 检查已保存 Cookie/Storage 有效性
- 有效 → 进入后台首页
- 失效 → 自动截图 + 通知人工扫码/验证码

### 阶段 2: 评价管理（写操作）

> 风控边界：系统定位为固定办公环境下的运营辅助工具，不做代理池、反指纹、CDP 隐藏或验证码/滑块绕过。真实写操作必须保留 dry-run、审批、审计、截图和熔断能力；差评举报和互动隐藏默认先生成候选建议，由运营确认后再执行。

- **1-3 星差评**: 读取评价 → AI/规则匹配举报建议 → 生成待审批候选动作 → 运营确认后 real-run 执行 → 截图留证
- **4-5 星好评**: 筛选未回复 → AI 选模板 + 填充变量 → 自动回复
- **负面互动**: 遍历【查看全部互动】列表 → AI/规则语义判断 → 生成待审批隐藏候选动作 → 运营确认后 real-run 隐藏 → 记录日志

### 阶段 3: 数据采集（只读）

| 指标类别 | 具体数据 | 来源 |
|----------|----------|------|
| 店铺健康度 | 综合体验星级、星级变化、严重劣质率、领航员综合行业排名、领航员未达标项(维度/考核指标/店铺表现/是否达标/下一星级标准)、平台求助率、3 分钟人工回复率、用户评价得分排名、积极评论率、成团-签收时效、物流综合违规处理率、店铺活跃度、消费者体验提升计划状态；传统 DSR(描述相符/服务态度/物流服务)仅在真实 DSR 标签出现时采集 | 服务数据 / 综合体验星级 |
| 评价数据 | 店铺评价分排名、近 30 天评价数、评价排名趋势、评价条数趋势 | 服务数据 / 评价数据 |
| 消费者体验 | 消费者服务体验总分、基础服务体验分、服务态度体验分、商品服务体验分、发货服务体验分、物流服务体验分、同行排名区间，以及较前 7 日变化；变化方向以页面 HTML 箭头为准，不从无符号文本猜测 | 服务数据 / 消费者体验指标 |
| 订单退款 | 纠纷退款数、纠纷退款率、介入订单数、平台介入率、品质退款率、平均退款时长、成功退款订单数、成功退款金额、成功退款率、退货退款自主完结时长、退款自主完结时长；纠纷退款率/平台介入率/品质退款率/平均退款时长输出趋势结论 | 服务数据 / 售后数据 |
| 订单申诉 | 申诉列表、状态、成功率 | 申诉中心 |

> 真实后台回归口径：综合体验星级页中的“近90天用户评价得分排名”“近30天3分钟人工回复率”“近30天成团-签收时效”是拼多多服务数据的 DSR 变种/领航员考核项，必须分别落到评价排名、回复率、签收时效等专用字段，不能写入传统 `dsrDesc`、`dsrService`、`dsrLogistics`。

### 阶段 4: 汇总写入
- 所有数据写入 DB
- 对比历史数据 → 标注异常波动
- 生成该店铺当日巡店摘要
- 所有店铺完成后聚合生成日报，并写入 `daily_reports` 形成可归档快照；日报 API 优先返回已归档版本，未完成或未归档时按当前数据动态生成。

## 四、报告格式

### 三层结构

| 层级 | 内容 | 用途 |
|------|------|------|
| 总览层 | 所有店铺按状态分组（正常/预警/异常） | 快速扫描 |
| 单店详情 | 指标趋势图 + 趋势结论 + 领航员未达标项 + 6 项消费者体验变化 + 操作明细 + 问题记录 | 深入分析 |
| 周报/月报 | 多店横向对比 + 排行 + 分布 | 管理层汇报 |

### 三色分级

- 🟢 正常: 指标稳定，无异常
- 🟠 预警: 指标连续下降或超过阈值
- 🔴 异常: 指标显著恶化，需人工介入

每店卡片显示: 星级变化、劣质率变化、操作摘要、AI 判定异常说明

## 五、数据模型（10 表）

|------|------|----------|
| stores | 店铺信息 & 凭证 | 店铺名、店铺ID、Cookie/Storage、状态、负责人、工厂关联、ai_config(JSON) |
| inspection_records | 巡店记录 | 店铺ID、日期、开始/结束时间、状态、耗时、Worker编号 |
| store_metrics | 每日指标快照 | 星级、劣质率、传统 DSR、领航员未达标项 JSON、评价数据、消费者体验分及变化、售后整体情况指标、申诉数与成功率 |
| review_actions | 评价操作明细 | 店铺ID、评价ID、星级、操作类型、内容、截图路径、状态 |
| interaction_actions | 互动处理记录 | 店铺ID、动态ID、内容摘要、AI判定、处理方式、状态 |
| reply_templates | 回复话术库 | 模板名、场景、内容、变量占位符、**store_id(NULL=全局)**、状态 |
| report_templates | 举报话术库 | 模板名、举报类型、话术、**store_id(NULL=全局)**、使用次数、成功率 |
| issues | 问题复盘 | 店铺ID、类型、严重度、描述、关联工厂、工厂反馈、整改状态 |
| users | 用户账户 | 用户名、密码哈希、角色(管理员/运营/工厂)、关联店铺/工厂 |

### 话术匹配逻辑: 全局 → 店铺级两级查找

```
AI 分析评价 → 优先匹配 store_id=当前店铺 的专属模板 → 未命中回退 store_id IS NULL 的全局模板
```

### 问题复盘 — 工厂协作

- **方式 A**: 按工厂筛选 → 导出 Excel
- **方式 B**: 生成带 token 的分享链接 → 工厂在线查看 + 更新整改状态

## 六、AI 介入点（5 个）

| # | 介入点 | 决策权 | 模型要求 |
|---|--------|--------|----------|
| 1 | 差评 → 举报话术匹配 | 自动执行 + 人工可覆审 | 轻量（分类+匹配） |
| 2 | 好评 → 回复模板选择 | 自动执行 | 轻量（分类+匹配） |
| 3 | 互动动态 → 负面判断 | 高置信自动 / 低置信标记人工 | 中量（语义判断） |
| 4 | 指标异常检测 | 标注+建议，不自动操作 | 轻量（数值分析） |
| 5 | 日报摘要生成 | 辅助生成，人工审核发布 | 重量（文本生成） |

### 模型调用策略

- 轻量任务 (介入点 1,2,4): 廉价模型高频调用，搭配规则引擎兜底
- 重量任务 (介入点 3,5): 强模型低频调用，结果可缓存可修改

### AI 安全边界

| 级别 | 范围 | 原则 |
|------|------|------|
| 🚫 绝不自动 | 修改店铺设置、删除商品、大额退款 | 无人工确认不执行 |
| ⚠️ 自动+审计 | 举报评价、回复好评、隐藏动态 | 截图留证 + 可回溯 |
| ✅ 自由发挥 | 数据采集、异常分析、报告生成 | 无破坏性 |

## 七、AI 模型可配置架构

### Provider 抽象层

定义统一的 `AIProvider` 接口（classify / matchTemplate / detectSentiment / detectAnomaly / generateSummary），底层可切换:

- ClaudeProvider (Anthropic Claude)
- OpenAIProvider (OpenAI GPT)
- DeepSeekProvider (DeepSeek)

### 三级配置粒度

| 层级 | 配置位置 | 优先级 |
|------|----------|--------|
| 全局默认 | .env / 系统设置 | 最低 |
| 店铺覆盖 | stores.ai_config (JSON) | 中 |
| 任务指定 | 代码中按介入点区分模型 | 最高 |

## 八、执行保障机制

### 三层重试 & 降级

| 失败场景 | L1 自动重试 | L2 降级处理 | L3 人工兜底 |
|----------|------------|------------|------------|
| 登录失效 | 刷新×3 + 切换Cookie备份 | 截图 + 通知扫码 | 运营重新登录 |
| 页面元素未找到 | 等超时延长 + 滚动 + 刷新 | 截图 + 跳过该条 | 人工补处理 |
| AI 低置信度 | 换prompt重试 + 强模型复核 | 标记待判断 + 安全默认策略 | 人工判定 |
| Worker 崩溃 | BullMQ 自动重分配 | 断点附近继续 | 连续崩溃暂停通知 |
| 风控拦截 | 立即暂停该店写操作并保存截图/HTML | 其他店继续读操作，必要时全局熔断写操作 | 人工接管处理安全验证或平台提示 |

### 核心原则

- **单条失败不阻塞整店**: 跳过继续，总完成率目标 ≥ 95%
- **AI 低置信度不盲动**: 宁可不做，不做错
- **实时告警**: 登录失效/连续失败/Worker崩溃 → 立即通知；完成率不足/低置信度 → 日报标注
- **完整性报告**: 每店铺输出执行完成率 + 未处理清单

## 十一、开发规范

### Git 版本管理
- 从 Phase 1 开始使用 Git
- 每个 Phase 完成打 tag: v0.1.0 → v0.2.0 → ... → v1.0.0
- CHANGELOG.md 记录每个版本的新增/修复/限制/测试覆盖

### 阶段测试
- 每个 Phase 完成后独立跑测
- 测试类型: 单元测试、集成测试、E2E(真实店铺)、AI准确率、执行完整性
- 测试报告写入 `docs/test-reports/phase-N-test.md`
- 报告模板包含: 环境、用例表、执行完整性、已知问题、AI准确率

### 关键风险
- PDD 后台页面结构变化 → 选择器需定期维护
- 风控策略 → 固定办公环境、低并发、写操作审批、异常熔断与人工接管；不做代理池、反指纹、CDP 隐藏或验证码/滑块绕过
- Cookie 有效期 → 监控 + 及时通知重登
