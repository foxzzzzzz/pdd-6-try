# 巡店结论输出实现说明与运营评审点

本文用于说明当前系统从“自动巡店”到“结论输出”的实际实现方式，以及在 LLM 接入有效或失败时，系统分别会生成怎样的结论。后续可同步给运营同学评审，再根据运营反馈调整输出模板和判断标准。

## 1. 当前实现链路

当前巡店结论输出已经形成最小闭环：

1. Worker 完成单店巡店，采集店铺指标、评价处理动作、互动处理动作和异常检测结果。
2. Worker 组装 `StoreReportData`：
   - `storeName`：店铺名
   - `metrics`：本次巡店采集到的指标字段
   - `reviewCount`：好评回复数量
   - `reportCount`：差评举报数量
   - `hideCount`：互动隐藏数量
   - `anomaly`：规则异常检测结果
   - `severity`：异常等级
3. 调用 `generateDailyReport([单店数据], store.aiConfig, useAI)` 生成摘要。
4. 调用 `formatDailySummaryForInspection()` 将结构化摘要拼成文本。
5. 写入 `inspection_records.summary`。
6. Web 展示：
   - 单店巡店详情/巡店列表展示 `inspection.summary`
   - `/reports/daily` 聚合所有店铺，展示总体摘要、关注店铺、建议事项和店铺明细
   - Dashboard 日报摘要卡跳转到完整日报页

## 2. LLM 接入有效时的 prompt 与预期输出

当 `WORKER_ENABLE_AI=true` 且 AI Provider/API key 可用时，摘要生成会调用 LLM。

当前传给 LLM 的 prompt 逻辑大致如下：

```text
根据以下 N 家拼多多店铺的巡店数据，生成日报摘要。
数据: [
  {
    "storeName": "店铺名",
    "metrics": { "...": "..." },
    "reviewCount": 0,
    "reportCount": 0,
    "hideCount": 0
  }
]

返回 JSON:
{
  "overview": "今日运营总体情况",
  "attentionStores": [{"name": "店铺名", "reason": "关注原因"}],
  "trends": "关键趋势分析",
  "recommendations": ["建议行动1", "建议行动2"]
}
```

系统提示要求 LLM：

- 以拼多多店铺运营助手身份分析数据
- 使用中文
- 只返回 JSON
- 不输出额外解释

预期 LLM 输出结构：

```json
{
  "overview": "今日巡店整体稳定，1 家店铺售后指标需关注。",
  "attentionStores": [
    {
      "name": "测试店铺",
      "reason": "平台介入率偏高，平均退款时长上升"
    }
  ],
  "trends": "售后核心指标中，平台介入率和平均退款时长需要重点关注。",
  "recommendations": [
    "优先复盘售后处理链路",
    "检查近期差评来源",
    "关注退款时长趋势"
  ]
}
```

写入单店巡店摘要后，预期展示为类似文本：

```text
今日巡店整体稳定，1 家店铺售后指标需关注。
趋势：售后核心指标中，平台介入率和平均退款时长需要重点关注。
关注：测试店铺(平台介入率偏高，平均退款时长上升)
建议：优先复盘售后处理链路；检查近期差评来源；关注退款时长趋势
```

## 3. LLM 接入失败时的规则 fallback

如果出现以下情况，系统会自动回退到规则模板：

- `WORKER_ENABLE_AI=false`
- API key 未配置
- Provider 调用失败
- LLM 返回内容不是合法 JSON
- LLM 返回字段为空

fallback 会根据本次巡店数据生成确定性结论：

- `overview`
  - 今日共巡店 X 家
  - X 家存在异常 / 全部正常
  - 自动回复好评 X 条，举报差评 X 条，处理互动 X 条
- `attentionStores`
  - `severity=warning` 或 `severity=critical` 的店铺
  - reason 优先使用异常检测描述，否则使用 severity
- `trends`
  - 无异常：各项指标暂无明显异常波动
  - 有异常：X 家店铺需要关注
- `recommendations`
  - 举报较多：建议复盘商品质量和评价来源
  - 有异常：建议人工复核异常店铺详细指标
  - 无异常：建议继续关注星级、体验分、售后核心指标变化

无异常时示例：

```text
今日共巡店1家，全部正常，自动回复好评0条，举报差评0条，处理互动0条。
趋势：各项指标暂无明显异常波动。
建议：整体运行稳定，继续关注星级、体验分、售后核心指标变化。
```

有异常时示例：

```text
今日共巡店1家，1家存在异常，自动回复好评0条，举报差评2条，处理互动1条。
趋势：1家店铺需要关注。
关注：某店铺(检测到3个异常指标：领航员未达标...)
建议：请人工复核1家异常店铺的详细指标。
```

## 4. 当前是否有固定输出模板

当前已经有两层模板：

1. LLM 输出 JSON schema：
   - `overview`
   - `attentionStores`
   - `trends`
   - `recommendations`
2. AI 失败后的规则 fallback 模板：
   - 总览
   - 关注店铺
   - 趋势
   - 建议

但当前还不是完整的运营日报模板。

目前更像“摘要卡 + 建议列表”，还没有固定成以下这种运营复盘结构：

```text
一、今日结论
二、风险店铺
三、售后核心指标
四、评价/互动处理
五、建议动作
六、需人工确认事项
```

## 5. 对当前准确性的判断

当前自动巡店对效率的帮助主要体现在：

- 自动采集核心指标
- 自动识别部分异常
- 自动生成初步摘要
- 自动聚合日报入口
- 帮运营快速定位“哪些店需要看”

但如果目标是判断“结论是否足以替代人工巡店复盘”，当前实现还需要进一步增强。

主要原因：

1. 当前传给 LLM 的 `metrics` 仍偏原始字段字典，不是经过业务语义重排后的关键指标摘要。
2. prompt 没有明确告诉 LLM 每个指标的业务方向，例如哪些指标升高是坏、哪些下降是坏。
3. prompt 没有明确阈值，例如平台介入率、品质退款率、平均退款时长达到什么水平需要预警。
4. fallback 模板只输出“正常/异常”和大致建议，不会完整展开售后 4 核心指标、评价趋势、消费者体验变化。
5. 当前结论没有强制要求“每条结论必须附带来源指标”，运营复核时可解释性还不够强。

## 6. 建议后续优化方向

为了让巡店结论真正提高运营复盘效率，建议下一步重点优化：

1. 将 LLM 输入从原始 `metrics` 改为业务结构化摘要。
2. 明确传入售后 4 核心指标：
   - 纠纷退款率
   - 平台介入率
   - 品质退款率
   - 平均退款时长
3. 明确传入评价数据：
   - 店铺评价分排名
   - 评价条数
   - 趋势判断
4. 明确传入消费者体验指标：
   - 消费者服务体验分
   - 服务态度体验分
   - 基础服务体验分
   - 商品服务体验分
   - 发货服务体验分
   - 物流服务体验分
   - 较前 7 日变化
5. prompt 中加入业务判断规则：
   - 哪些指标越高越差
   - 哪些指标越低越差
   - 什么阈值进入 warning/critical
6. 输出固定运营模板。
7. 每条结论附带来源指标，例如：
   - “平台介入率 0.65%，趋势上升”
   - “平均退款时长 6.57 小时，趋势上升”
   - “领航员未达标：近30天3分钟人工回复率 52.17%，目标 75%”
8. fallback 模板也展示核心指标，而不是只输出“正常/异常”。

## 7. 当前结论

当前“巡店 -> 结论输出”已经具备可用闭环，但结论质量仍偏初步摘要。

如果只是帮助运营快速定位风险店铺，当前实现已经有价值。

如果目标是显著替代人工巡店复盘，需要继续优化 prompt、业务结构化输入和固定日报模板，并通过真实巡店样本让运营评估结论准确率。
